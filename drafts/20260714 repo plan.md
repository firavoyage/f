# how is it to have readme of different localizations on repo root or docs

Managing localized README files (e.g., `README.md`, `README.zh.md`) on your repository root versus placing them inside a dedicated `docs/` folder depends on your project's audience and hosting platform.

Here is a breakdown of how both approaches work, along with their pros and cons.

### Approach 1: Multiple READMEs in the Repository Root

You place all language versions directly in the root directory using language suffixes.

- **Examples:** `README.md` (English), `README.zh-CN.md` (Simplified Chinese), `README.ja.md` (Japanese).

### Pros

- **High Visibility:** Users instantly see the available language options when landing on the repository main page.
- **Platform Support:** Platforms like GitHub and GitLab automatically render these files if users click on them.
- **Easy Maintenance:** Small fixes can be made directly in the root without navigating deep folder structures.

### Cons

- **Root Clutter:** Having 5 to 10 README files crowds the root directory, making it look messy.
- **No Auto-Detection:** GitHub will only render the default `README.md` automatically; users must manually click the other files to read them.
- **Clunky Navigation:** You must manually add a language switcher menu (links) at the top of every README file.

***

### Approach 2: Localization Inside a `docs/` Folder

You keep only the primary README in the root and move all localized versions into a subfolder.

- **Examples:** `docs/README.zh.md` or a structured path like `docs/languages/zh/README.md`.

### Pros

- **Clean Root:** Keeps your repository root tidy, professional, and focused on core code configuration files.
- **Static Site Integration:** Perfect if you use documentation generators like Docusaurus, MkDocs, or Sphinx, which natively turn these folders into a multilingual website.
- **Scalability:** Allows you to translate entire documentation suites, not just a single introduction file.

### Cons

- **Lower Discoverability:** Foreign-language users might not realize translations exist unless you prominently link to them from the root README.
- **Extra Clicks:** Users must click through folders on GitHub to read the documentation in their language.

***

### Best Practices for Implementation

If you choose the **Repository Root** approach:

- Use standard ISO language codes (e.g., `zh`, `es`, `ja`).
- Put a clear, prominent language selection menu at the very top of your main `README.md`.

If you choose the **Docs Folder** approach:

- Add a "Documentation in other languages" section near the top of your root `README.md` with direct links to the localized files.

# compare localization folder structure like docs/foo/readme.jp.md or docs/jp/foo/readme.md

When structuring localized Markdown documentation, choosing between **file-suffix localization** (`docs/foo/readme.jp.md`) and **directory-level localization** (`docs/jp/foo/readme.md`) depends on your project size, translation tooling, and hosting platform.

Here is a comprehensive comparison to help you choose the best strategy.

### Overview Comparison

| Metric / Feature              | File Suffix (`docs/foo/readme.jp.md`)             | Directory Level (`docs/jp/foo/readme.md`)              |
| ----------------------------- | ------------------------------------------------- | ------------------------------------------------------ |
| **Primary Structural Focus**  | Subject/Topic-first                               | Language-first                                         |
| **Best For**                  | Small projects, 1–2 languages, single maintainers | Large docs sites, multiple languages, localized assets |
| **Static Site Generator Fit** | Hugo, custom scripts                              | Docusaurus, Nextra, Fern                               |
| **Asset Management**          | Shared assets folder                              | Duplicate or relative path complexities                |
| **GitHub Navigation**         | Easy context switching per folder                 | Requires navigating out of the language tree           |

***

### Strategy 1: File Suffix (`docs/foo/readme.jp.md`)

This strategy groups all linguistic variants of a specific document inside the same topic folder.

- **Proximity**: Keeps the source language and all target translations in the exact same directory, making it highly obvious when a translation needs updating.
- **Asset Sharing**: Image files, diagrams, and localized snippets can sit in `docs/foo/` and be linked via simple relative paths (`./diagram.png`) across all language versions.
- **Cluttered Folders**: If you translate your documentation into 10+ languages, a single folder will contain dozens of files, making manual navigation messy.
- **Tooling Friction**: Many modern documentation platforms expect a distinct directory tree per locale and struggle to parse inline file flags natively. [link](https://discourse.gohugo.io/t/folder-structure-for-multi-language/20582)

***

### Strategy 2: Directory Level (`docs/jp/foo/readme.md`)

This strategy isolates each language into its own absolute root directory. [link](https://buildwithfern.com/learn/docs/localization/overview)

- **Ecosystem Standards**: This is the industry standard for major Static Site Generators (SSGs) like Docusaurus or [Fern Docs](https://buildwithfern.com/learn/docs/localization/overview). They automatically map `docs/jp/` to the `/jp/` URL slug.
- **Access Control & Isolation**: Allows you to safely hand off the `docs/jp/` folder to localized translation teams or external vendors without them touching other language tracks.
- **Drift Risk**: Because the files are far apart, it is much easier for the translated files to drift out of sync with the main codebase as features change.
- **Asset Duplication**: References to shared files (like `../../assets/image.png`) require complex relative tracking, or you must duplicate images inside every language folder. [link](https://www.reddit.com/r/github/comments/1r1jwcx/why%5Fdo%5Fmany%5Fgithub%5Foss%5Fprojects%5Fskip%5Freadme%5Fdocs/)

***

### Recommendation Matrix

- Choose **File Suffix (`readme.jp.md`)** if you are managing a small GitHub repository, have only one or two translation tracks, and rely on standard GitHub file viewing rather than a dedicated documentation website.
- Choose **Directory Level (`docs/jp/...`)** if you plan to scale past 3 languages, use a Translation Management System (TMS) like Crowdin or Lokalise, or deploy a dedicated user-facing documentation portal. [link](https://github.com/orgs/community/discussions/50719)

To help you choose, what **static site generator or framework** (e.g., Docusaurus, Hugo, Next.js) are you using to host your documentation?

## pros and cons

Here is the direct breakdown of the pros and cons for each localization folder structure.

### File Suffix Strategy (`docs/foo/readme.jp.md`)

### Pros

- **High Visibility**: Translators see immediately if a translation is missing next to the original.
- **Easy Asset Sharing**: Shared images use simple relative paths (`./image.png`) in the same folder.
- **Simple Renaming**: Moving or renaming a topic folder automatically moves all its language variants.
- **Fewer Directories**: Keeps the overall folder tree shallow and easy to navigate in an IDE.

### Cons

- **Folder Clutter**: Translating into 5+ languages fills a single folder with dozens of files.
- **SSG Incompatibility**: Many modern documentation platforms cannot parse suffix flags out of the box.
- **No Isolated Access**: You cannot easily restrict a vendor's write access to just one language.
- **Build Complexities**: Requires custom regex or scripts to filter and route files during deployment.

***

### Directory Level Strategy (`docs/jp/foo/readme.md`)

### Pros

- **Industry Standard**: Native structure for major platforms like Docusaurus, Nextra, and Astro.
- **Clean URLs**: Automatically maps the folder tree to logical URL paths like `://site.com`.
- **Vendor Isolation**: Localization teams can be granted Git access strictly to their language folder.
- **Scales Well**: Handles 20+ languages cleanly without cluttering the core development folders.

### Cons

- **High Out-of-Sync Risk**: The root file and translation live far apart, making content drift easy.
- **Asset Complexities**: Requires long, brittle relative paths (`../../assets/pic.png`) or duplicated files.
- **Heavy Maintenance**: Structural changes in the main language must be manually duplicated across all trees.
- **Deep Nesting**: Developers must drill down deep into multiple identical sub-folders to check text.

***

## pros and cons if engineering is trivial

If we assume engineering implementation is trivial—meaning custom deployment scripts, URL routing, asset linking, and framework compatibility are non-issues—the decision comes down purely to **human workflow, content maintenance, and collaboration.**

Here are the pros and cons of each structure when engineering friction is removed:

### File Suffix Strategy (`docs/foo/readme.jp.md`)

### Pros

- **Immediate Context**: Writers see the source file and all translations side-by-side, preventing them from forgetting to update a localized file when editing the original.
- **Low Structural Overhead**: If a feature changes and a folder needs to be renamed or deleted, it happens once for all languages simultaneously.
- **Unified Discussions**: PR reviews for content changes naturally group all language updates into the same file-system location, making code reviews cohesive.

### Cons

- **Cognitive Noise for Devs**: Engineers browsing the codebase will see a wall of `.md` files for languages they don't read, making the repo feel messy.
- **Granular Permission Limits**: You cannot easily use Git permissions (like GitHub `CODEOWNERS`) to restrict external translators to only edit their specific language.
- **Merge Conflict Risk**: Concurrent edits by multiple language translators on the same directory block can cause frequent repository merge conflicts.

***

### Directory Level Strategy (`docs/jp/foo/readme.md`)

### Pros

- **Clean Codebase**: Engineers only see the source language in their core workspace, keeping the primary development environment clutter-free.
- **Total Vendor Isolation**: You can safely grant specific localization agencies or internal regional teams write access exclusively to their language directory tree.
- **Independent Shipping**: Fast-moving language tracks can merge changes and push updates completely independent of slower translation tracks.

### Cons

- **Severe Content Drift**: Out of sight, out of mind. Writers will update the primary language root, and the localized directories will quietly fall out of date without anyone noticing.
- **Massive Structural Fragility**: If an engineer deletes or moves a folder in the English directory, they must manually replicate that structural change across every single language directory.
- **Contextless PRs**: Reviewing a translation PR requires jumping completely out of the current folder tree to verify if the file structure still matches the source language.

***

If engineering overhead doesn't matter, choose **File Suffix** if your priority is keeping content accurately in sync, or **Directory Level** if your priority is isolating external translators.

## i think file suffix is always better as pages can have different translations like wikipedia, and everything else like engineering or human stuff are trivial

That is a highly valid perspective. If you treat documentation exactly like Wikipedia—where articles exist as independent linguistic entities that grow, change, and diverge naturally over time—the **File Suffix** structure is structurally superior.

When you remove technical limitations and assume human friction is trivial, your approach offers two massive advantages that the directory-level structure completely breaks:

### 1. True 1:1 Domain Mapping

In Wikipedia, a page about a topic is tied to that specific topic, not to a language silo. Using file suffixes keeps the entity unified. If you want to check if the "Authentication API" page has a Japanese or French version, you look at the exact same folder. You don't have to jump across entirely different directory trees to see what exists.

### 2. Graceful Asymmetric Growth

Just like Wikipedia, translation isn't always a 100% accurate mirror.

- The Japanese page might need a massive section explaining a regional compliance law (like Japan's APPI).
- The English page doesn't need that section at all.
- With a file suffix (`auth.jp.md`), that specific page can expand, morph, and host its own regional nuances right alongside the core file, without forcing you to maintain an entire shadow directory tree just for one unique page.

### The One True Breakpoint: When it Fails

The only time the Wikipedia model collapses under a file-suffix system is **when you don't actually want a Wikipedia model**.

If your goal is an **Enterprise Product Manual** rather than a Wiki, you usually want strict, 100% structural parity. If the English manual updates to version 2.0, you _want_ the Japanese page to be forced to update or explicitly show a "version out of date" warning. File suffixes make it incredibly easy for pages to accidentally diverge into completely different content, destroying the "single source of truth" aspect that product manuals require.

But if your goal is an open, organic, and highly adaptable knowledge base where languages are allowed to evolve independently, you are absolutely right: the file suffix is structurally the most logical way to group information.

## i think file suffix always better even for documentation

If you view a documentation site through the lens of a **unified, topic-centric graph**, the file suffix approach (`docs/foo/readme.jp.md`) offers an elegance that directory isolation can never match. It treats language as a property of the file rather than a property of the universe.

Even in strict product documentation, this approach provides two massive structural advantages:

### 1. Atomic Content Lifecycle

In a file-suffix system, a documentation topic is treated as a single, atomic unit.

- **The Problem with Directories:** If a feature is deprecated and deleted, an engineer must delete `docs/en/foo`, `docs/jp/foo`, `docs/es/foo`, and so on. If they miss one, you get ghost pages.
- **The File Suffix Solution:** When the feature is deleted, you delete the `docs/foo/` folder. The source material, the Japanese translation, and the Spanish translation all vanish from the codebase in a single, atomic git commit. Clean up is perfect.

### 2. Perfect Fallback Architecture

Documentation sites often suffer from "partial translation"—where only 40% of the site is localized.\
With file suffixes, building a fallback mechanism is incredibly clean. If a user requests `docs/foo/readme.md` in Japanese, the rendering engine looks for `readme.jp.md`. If it doesn't exist, it instantly falls back to the default `readme.md` in the exact same directory. There is no need to crawl up and down different directory trees to find a matching file structure.

***

### The Ultimate Trade-Off: Git Scale vs. Conceptual Cleanliness

While conceptually superior, the reason many massive tech companies (like Microsoft, Google, or AWS) abandon the suffix model for their core documentation isn't due to site generation—it is due to **Git metadata pollution at scale**.

If you have 100 developers editing documentation daily, and 20 translation vendors pushing automated updates nightly:

- **In a Directory System:** Vendors commit exclusively to `/docs/jp/`. Developers commit to `/docs/en/`. Their Git history, pull requests, and commit logs never intersect.
- **In a Suffix System:** Every single folder becomes a high-traffic intersection. A developer trying to review the git history (`git log docs/foo/`) to see how a feature evolved will find the history heavily polluted with hundreds of localization machine-commits.

If you have a tight-knit team or automated tooling that can filter out git noise, **file suffix wins on conceptual purity every time.** It keeps the domain model of your documentation organized by _what_ it is, rather than _how_ it is spoken.
