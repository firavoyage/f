# how is it to have readme of different localizations on repo root or docs

Managing localized README files (e.g., `README.md`, `README.zh.md`) on your repository root versus placing them inside a dedicated `docs/` folder depends on your project's audience and hosting platform.

Here is a breakdown of how both approaches work, along with their pros and cons.

### Approach 1: Multiple READMEs in the Repository Root

You place all language versions directly in the root directory using language suffixes.

- **Examples:** `README.md` (English), `README.zh-CN.md` (Simplified Chinese), `README.ja.md` (Japanese).

### Pros

- **High Visibility:** Users instantly see the available language options when landing on the repository main page.
- **Platform Support:** Platforms like GitHub and GitLab automatically render these files if users click on them.
- **Easy Maintenance:** Small fixes can be made directly in the root without navigating deep folder structures.

### Cons

- **Root Clutter:** Having 5 to 10 README files crowds the root directory, making it look messy.
- **No Auto-Detection:** GitHub will only render the default `README.md` automatically; users must manually click the other files to read them.
- **Clunky Navigation:** You must manually add a language switcher menu (links) at the top of every README file.

***

### Approach 2: Localization Inside a `docs/` Folder

You keep only the primary README in the root and move all localized versions into a subfolder.

- **Examples:** `docs/README.zh.md` or a structured path like `docs/languages/zh/README.md`.

### Pros

- **Clean Root:** Keeps your repository root tidy, professional, and focused on core code configuration files.
- **Static Site Integration:** Perfect if you use documentation generators like Docusaurus, MkDocs, or Sphinx, which natively turn these folders into a multilingual website.
- **Scalability:** Allows you to translate entire documentation suites, not just a single introduction file.

### Cons

- **Lower Discoverability:** Foreign-language users might not realize translations exist unless you prominently link to them from the root README.
- **Extra Clicks:** Users must click through folders on GitHub to read the documentation in their language.

***

### Best Practices for Implementation

If you choose the **Repository Root** approach:

- Use standard ISO language codes (e.g., `zh`, `es`, `ja`).
- Put a clear, prominent language selection menu at the very top of your main `README.md`.

If you choose the **Docs Folder** approach:

- Add a "Documentation in other languages" section near the top of your root `README.md` with direct links to the localized files.

a page can have many localization (e.g. wikipedia simple english). a language could not have specicial dedicate pages. (regional promotional stuff are irrelevant.)
