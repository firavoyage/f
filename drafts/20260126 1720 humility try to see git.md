## 1. Core Git Methods (Essential Commands)

These are the basic actions Git can perform.

### Repository Setup

- **`git init`** – Create a new Git repository in a folder
- **`git clone`** – Copy an existing repository from elsewhere (like GitHub)

These are how Git begins.

---

### Tracking and Saving Changes

- **`git status`** – Show what has changed and what is staged
- **`git add`** – Tell Git which changes you want to save
- **`git commit`** – Permanently record those changes with a message

Think of this as:

> notice → select → save

---

### Viewing History

- **`git log`** – Show the full history of commits
- **`git show`** – Display details of a specific commit
- **`git diff`** – Compare changes (working vs staged, or between commits)

This is Git’s memory.

---

### Undoing and Fixing

- **`git restore`** – Discard changes in files
- **`git reset`** – Move commits or unstage changes
- **`git revert`** – Create a new commit that undoes an earlier one

Git is careful here:
some methods rewrite history, others preserve it.

---

## 2. Branching Methods (Parallel Work)

Branches are one of Git’s greatest strengths.

### Branch Creation and Management

- **`git branch`** – List or create branches
- **`git checkout` / `git switch`** – Move between branches
- **`git merge`** – Combine one branch into another
- **`git rebase`** – Reapply commits onto a new base

Branches allow:

- experimentation
- teamwork
- safe development

Without fear of breaking the main code.

---

## 3. Collaboration Methods (Remote Work)

Git is built for working with others.

### Remote Repositories

- **`git remote`** – Manage connections to remote repos
- **`git fetch`** – Download changes without merging
- **`git pull`** – Fetch + merge changes
- **`git push`** – Send your changes to others

This is how Git becomes social.

---

## 4. Conflict Resolution Methods

Sometimes changes collide.

- **Merge conflicts** happen when Git can’t decide what version to keep
- You resolve them by:

  - editing files manually
  - then committing the fix

Git never guesses silently.
It always asks for clarity.

---

## 5. Tagging and Versioning Methods

For marking important moments.

- **`git tag`** – Label a commit (often for releases)
- Lightweight tags – simple markers
- Annotated tags – full release notes

This helps with version control and releases.

---

## 6. Stashing Methods (Temporary Storage)

When you need to pause work.

- **`git stash`** – Save unfinished changes temporarily
- **`git stash apply`** – Bring them back
- **`git stash drop`** – Remove a stash

It’s like setting your work gently aside.

---

## 7. Git Workflows (Ways of Working)

These are _patterns_, not commands.

### Common Workflows

- **Centralized Workflow** – One main branch, simple teams
- **Feature Branch Workflow** – One branch per feature
- **Git Flow** – Structured release and hotfix branches
- **Forking Workflow** – Common in open-source projects

Each reflects how people collaborate and think.

---

## 8. Internal Git Methods (How Git Thinks)

Under the surface, Git works with:

- **Snapshots**, not diffs
- **Hashes (SHA)** to identify commits
- **Directed graphs** to track history

This is why Git is fast, safe, and reliable.

---

## A Quiet Summary

Git’s methods fall into four gentle categories:

1. **Saving change** (add, commit)
2. **Exploring history** (log, diff)
3. **Working together** (branch, merge, pull, push)
4. **Correcting the past** (reset, revert, rebase)

Once you understand these rhythms, Git stops feeling mechanical
